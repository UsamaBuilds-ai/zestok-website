    - Request: GET /api/pin/verify with header `x-access-pin: wrong`
    - Prerequisite: app_settings has key='pin' with bcrypt hash
    - Response: HTTP 401, `{ valid: false, message: 'Invalid PIN' }`

    - Request: GET /api/pin/verify with any header
    - Prerequisite: app_settings table is empty or lacks key='pin'
    - Response: HTTP 200, `{ configured: false, valid: false }`
    
    **Case 4 — No PIN header:**
    - Request: GET /api/pin/verify without x-access-pin header
    - Prerequisite: app_settings has key='pin'
    - Response: HTTP 401, `{ valid: false, message: 'PIN required' }`
    
    **Case 5 — Rate limit exceeded:**
    - Request: 6 rapid GET /api/pin/verify requests with wrong PIN
    - Response (6th): HTTP 429, `{ error: 'Too many PIN attempts. Try again later.' }`
    
    **Case 6 — Database unreachable:**
    - Request: GET /api/pin/verify while PostgreSQL is down
    - Response: HTTP 503, `{ error: 'database_unreachable', message: '...' }`
  </behavior>

    1. Require `bcrypt` and `express-rate-limit`
    2. Create PIN rate limiter: windowMs=15*60*1000, limit=5, draft-8 headers
    3. Create async middleware `verifyPin` that:
       - Queries app_settings for key='pin'
       - Returns `{ configured: false, valid: false }` HTTP 200 if no pin row exists (D-11)

       - Uses `bcrypt.compare()` against stored hash

       - Catches DB errors to return HTTP 503 (D-09)
    4. Apply rate limiter + verifyPin to GET /api/pin/verify route

    Update src/server.js to add PIN verification with bcrypt and express-rate-limit.

    1. **Add requires** at the top of server.js:
       ```javascript
       const bcrypt = require('bcrypt');
       const rateLimit = require('express-rate-limit');
       ```

    2. **Create rate limiter** for PIN verify endpoint (per D-08):
       ```javascript
       const pinVerifyLimiter = rateLimit({
         windowMs: 15 * 60 * 100,  // 15 minutes (in milliseconds)
         limit: 5,                  // 5 attempts per window
         standardHeaders: 'draft-8',
         legacyHeaders: false,
         message: { error: 'Too many PIN attempts. Try again later.' }
       });
       ```
       Use `limit` (not deprecated `max`) per RESEARCH.md Pitfall 4. The 5/15min window provides reasonable protection without excessive user friction for a local-network device.

    3. **Create `verifyPin` middleware** (async, wraps the PIN check):
       ```javascript
       const verifyPin = async (req, res, next) => {
         try {
           const result = await pool.query(
             "SELECT value FROM app_settings WHERE key = 'pin'"
           );

           // No PIN configured (D-05, D-07, D-11)
           if (result.rows.length === 0 || !result.rows[0].value) {
             return res.status(200).json({ configured: false, valid: false });
           }

           const userPin = req.headers['x-access-pin'];

           // No PIN header provided (D-10 variant)
           if (!userPin) {
             return res.status(401).json({ valid: false, message: 'PIN required' });
           }

           // bcrypt max input guard (Pitfall 2 — 72 byte limit)
           if (userPin.length > 72) {
             return res.status(401).json({ valid: false, message: 'Invalid PIN' });
           }

           const storedHash = result.rows[0].value;
           const isValid = await bcrypt.compare(userPin, storedHash);

           if (!isValid) {
             return res.status(401).json({ valid: false, message: 'Invalid PIN' });
           }

           next(); // PIN is valid — proceed to route handler
         } catch (err) {
           console.error('PIN verification error:', err.message);
           return res.status(503).json({
             error: 'database_unreachable',
             message: 'Database is not available.'
           });
         }
       };
       ```

    4. **Add GET /api/pin/verify route** with rate limiter AND verifyPin:
       ```javascript
       app.get('/api/pin/verify', pinVerifyLimiter, verifyPin, (req, res) => {
         res.json({ valid: true });
       });
       ```
       Note: The rate limiter is applied to this specific route only, per RESEARCH.md Anti-Patterns ("Rate limiting the wrong endpoint"). Stock queries are read-only and don't need rate limiting.

    5. **Protect GET /api/stock** with verifyPin middleware:
       ```javascript
       app.get('/api/stock', verifyPin, async (req, res) => {
         // Existing mock code — replaced in Plan 01-03
       });
       ```
       This temporarily protects /api/stock with the real middleware. The mock data stays until Plan 01-03.

    6. **Update existing GET /api/pin/status** — add DB error handling matching D-09 pattern:
       Ensure it also catches pool.query errors with try/catch and returns 503.

    **Why bcrypt.compare() and not a custom comparison:**
    bcrypt.compare() is timing-attack-resistant (RESEARCH.md Anti-Patterns — "Don't hand roll" PIN hashing). The compare function extracts the salt from the stored hash, hashes the input, and compares — all in constant time relative to the hash output length. A custom comparison would leak timing information.

    **Why express-rate-limit and not a custom counter:**
    express-rate-limit is the standard Express middleware for rate limiting (RESEARCH.md "Don't Hand Roll" table). It handles IP tracking, window management, standard headers, and is purpose-built for Express routes. A custom implementation would miss edge cases (race conditions, IP rotation, header edge cases).

    **IMPORTANT AVOIDANCES:**
    - Do NOT log the PIN value — ever (D-06: plaintext never persisted). log only `'PIN verification error'` with `err.message`.
    - Do NOT return the PIN or hash in any response body.
    - Do NOT use `max` option for rate limiter — use `limit` (RESEARCH.md Pitfall 4: express-rate-limit renamed `max` to `limit` in v7).
    - Do NOT apply rate limiter to /api/pin/status or /api/stock — only the verify endpoint needs it (RESEARCH.md Anti-Patterns).
    - Do NOT string-interpolate the x-access-pin value into SQL — use parameterized queries (the pin value is not used in SQL directly in this case, but the principle extends to any route that does use user input in queries).
    - Do NOT add the bcrypt hash literal in server.js — the hash comes from the database (set by desktop Phase 2). The only hash operation in Phase 1 is `bcrypt.compare()` (verify), not `bcrypt.hash()` (set).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Mobile client → Express API | x-access-pin header crosses network boundary; untrusted input from LAN |
| Express API → PostgreSQL | bcrypt hash read from app_settings over local TCP |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Denial of Service | GET /api/pin/verify — brute force | mitigate | express-rate-limit: 5 attempts per 15-min window per IP (D-08). Responds with HTTP 429 + JSON error body after limit exceeded |
| T-01-05 | Information Disclosure | PIN comparison — timing attack | mitigate | bcrypt.compare() is timing-safe by design — compares full digest, not input string length (RESEARCH.md Security Domain: "Timing attack on PIN comparison") |
| T-01-06 | Information Disclosure | PIN in application logs | mitigate | Never log pin value; log only error context (err.message, not req.headers['x-access-pin']). Console.error uses generic messages ('PIN verification error') |
| T-01-07 | Tampering | bcrypt max input bypass (72+ byte PIN) | mitigate | Explicit length check before bcrypt.compare(): reject >72 byte inputs with 401 (RESEARCH.md Pitfall 2) |
</threat_model>
