    1. **Locate the existing GET /api/stock route** in server.js (currently has verifyPin middleware from Plan 01-02, and a mock handler returning hardcoded data).

    2. **Replace the handler function** with:
       ```javascript
       app.get('/api/stock', verifyPin, async (req, res) => {
         try {
           const result = await pool.query(
             `SELECT item, category, in_qty, out_qty, balance, latest_rate, value
              FROM stock_balance
              WHERE balance > 0
              ORDER BY category, item`
           );
           res.json(result.rows);
         } catch (err) {
           console.error('Stock query error:', err.message);
           res.status(503).json({
             error: 'database_unreachable',
             message: 'Database is not available.'
           });
         }
       });
       ```

       **What this query does:**
       - Queries the stock_balance SQL view (defined in src/db/schema.sql per DB-04)
       - `WHERE balance > 0` filters out depleted items (per API-03)
       - `ORDER BY category, item` groups by category first, then sorts by item name (per API-02)
       - The view already includes category per item (per API-06)
       - Column aliases use snake_case (in_qty, out_qty, latest_rate) matching the view definition
       - verifyPin middleware (added in Plan 01-02) protects the endpoint and returns { configured: false } if no PIN set

       **Column mapping from view to API response:**
       | SQL View Column | API Response Key | Source |
       |----------------|------------------|--------|
       | item | item | stock_entries.item |
       | category | category | stock_entries.category |
       | in_qty | inQty | Computed: SUM of 'in' entries |
       | out_qty | outQty | Computed: SUM of 'out' entries |
       | balance | balance | Computed: in - out |
       | latest_rate | latestRate | MAX rate of 'in' entries |
       | value | value | balance * latest_rate |

       The response uses `result.rows` directly from pg, which returns objects with camelCase-converted keys? No — pg returns snake_case exactly as defined in the view. The fields in response will be `in_qty`, not `inQty`. 

       **This is the standard behavior of pg** — it doesn't transform column names. The API response will send snake_case to the mobile client. This is acceptable and common for REST APIs. The mobile app (Phase 3) should consume snake_case keys. Alternatively, if the mobile team expects camelCase, we'd need a transformation layer (mapping query or middleware).

       **Decision:** Keep snake_case from the view in the API response. It's standard REST practice, and the mobile app can use the keys directly. If needed, a JSON response transformation can be added in Phase 3. This avoids adding mapping complexity in Phase 1.

    3. **Remove any remaining mock code**: Ensure `getStockData()`, hardcoded mock arrays, and any temporary placeholder data are fully removed from server.js.

  <action>
    Create `src/db/migrate.js` — a standalone Node.js script that reads the existing `stock-data.json` file and inserts all entries into PostgreSQL.

    This follows the pattern in RESEARCH.md Code Examples (lines 452-536) with these specifics:

    **Script structure:**
    ```javascript
    require('dotenv').config();
    const fs = require('fs/promises');
    const path = require('path');
    const pool = require('./pool');

    const MIGRATION_KEY = 'data_migration_complete';
    ```

    **Migration logic:**

    1. **Idempotency check:** Query `app_settings WHERE key = 'data_migration_complete'`. If value is 'true', log "Migration already completed. Skipping." and exit.

    2. **JSON file location** (per RESEARCH.md Open Question #1):
       - Accept `USER_DATA_PATH` environment variable override
       - Fallback: `process.env.APPDATA + '/stock-management/stock-data.json'`
       - On Windows, `process.env.APPDATA` is typically `C:\Users\<user>\AppData\Roaming`
       - Handle ENOENT gracefully: log "No stock-data.json found. Skipping migration." and exit

    3. **Parse JSON:** Read the file, parse as JSON. Expect `{ entries: [...] }` structure matching the current CreateStore (from src/main.js createStore).

    4. **Transactional insert** (per DB-05 — atomic data migration):
       ```javascript
       const client = await pool.connect();
       try {
         await client.query('BEGIN');
         
         for (const entry of entries) {
           await client.query(
             `INSERT INTO stock_entries (id, date, type, item, category, quantity, rate, note, created_at)
              VALUES ($1, $2, $3, $4, $5, $6, $7, $8, $9)
              ON CONFLICT (id) DO NOTHING`,
             [entry.id, entry.date, entry.type, entry.item, entry.category || '',
              entry.quantity, entry.rate, entry.note || '', entry.createdAt]
           );
         }
         
         // Mark migration complete
         await client.query(
           `INSERT INTO app_settings (key, value) VALUES ($1, 'true')
            ON CONFLICT (key) DO UPDATE SET value = 'true', updated_at = NOW()`,
           [MIGRATION_KEY]
         );
         
         await client.query('COMMIT');
         console.log(`Migration complete: ${entries.length} entries imported.`);
       } catch (err) {
         await client.query('ROLLBACK');
         throw err;
       } finally {
         client.release();
       }
       ```

    5. **Exit handling:** Call `pool.end()` before process exit to allow Node.js to terminate naturally.

    6. **Main entry point** (script execution):
       ```javascript
       async function migrate() { /* ... */ }
       
       migrate().catch(err => {
         console.error('Migration failed:', err);
         process.exit(1);
       });
       ```

    **Field mapping from JSON (entry model) to PostgreSQL (stock_entries):**
    | JSON field | SQL column | Notes |
    |------------|-----------|-------|
    | entry.id | id | UUID — ON CONFLICT DO NOTHING for idempotency |
    | entry.date | date | ISO date string |
    | entry.type | type | 'in' or 'out' — matches CHECK constraint |
    | entry.item | item | String |
    | entry.category | category | Default '' if missing |
    | entry.quantity | quantity | Numeric |
    | entry.rate | rate | Numeric |
    | entry.note | note | Default '' if missing |
    | entry.createdAt | created_at | ISO timestamp string → TIMESTAMPTZ |

    **Why a standalone script (not part of server startup):**
    The migration should be a one-time, manually-invoked operation (RESEARCH.md "Specific Ideas" — "called manually or as part of app startup when DB is empty"). Keeping it standalone gives the user control over when migration happens. The `npm run migrate` script (added in Plan 01-01 Task 1) makes invocation easy: `npm run migrate`.

    **Why transactional with ON CONFLICT:**
    - Transaction ensures all-or-nothing: if any insert fails, the entire migration rolls back without partial data
    - `ON CONFLICT (id) DO NOTHING` makes the script idempotent — running it multiple times won't create duplicate rows
    - Tracking migration status in `app_settings` prevents unnecessary re-runs

    **IMPORTANT AVOIDANCES:**
    - Do NOT store the migration flag file on disk — track it in app_settings table (DB-backed state is queryable and survives app reinstall)
    - Do NOT delete the stock-data.json after migration — the desktop app (Phase 2) still uses JSON file storage; deletion would break Phase 2 workflows
    - Do NOT assume entries array is non-empty — handle empty gracefully with log message
    - Do NOT use `pool.query()` inside the loop without a transaction — each INSERT would be autocommitted; use a client with BEGIN/COMMIT for atomicity
    - Do NOT hardcode the JSON path — use env var with documented fallback (RESEARCH.md Open Question #2)
    - Do NOT require the JSON file to exist — handle ENOENT by logging and skipping (graceful migration for clean installs)
    - Do NOT `require('../../src/db/pool')` from within src/db/ — use `require('./pool')` since migrate.js lives in the same directory as pool.js
