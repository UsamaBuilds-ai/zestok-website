       - Import `Pool` from `pg`

       **Why Pool not Client:** Pool handles concurrent requests, connection lifecycle, and error recovery automatically. A single Client serializes all queries and blocks under load. (RESEARCH.md Anti-Patterns: "Directly using pg Client instead of Pool")

         - id UUID PRIMARY KEY
         - date DATE NOT NULL
         - type VARCHAR(3) NOT NULL CHECK (type IN ('in', 'out'))
         - item VARCHAR(255) NOT NULL
         - category VARCHAR(255) NOT NULL DEFAULT ''
         - quantity NUMERIC(12,2) NOT NULL CHECK (quantity > 0)
         - rate NUMERIC(12,2) NOT NULL CHECK (rate >= 0)
         - note TEXT NOT NULL DEFAULT ''
         - created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
         - Indexes on item, category, date, type (matching common query patterns from renderer.js)
       
       - `app_settings` table (key-value store per DB-03):
         - key VARCHAR(255) PRIMARY KEY
         - value TEXT NOT NULL DEFAULT ''
         - created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
         - updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
       
       - `stock_balance` view (computed from stock_entries per DB-04, mirrors renderer.js getBalances() lines 22-51):
         - item, category
         - in_qty: SUM(CASE WHEN type = 'in' THEN quantity ELSE 0 END)
         - out_qty: SUM(CASE WHEN type = 'out' THEN quantity ELSE 0 END)
         - balance: SUM(CASE WHEN type = 'in' THEN quantity ELSE -quantity END)
         - latest_rate: MAX(CASE WHEN type = 'in' THEN rate ELSE 0 END) FILTER (WHERE type = 'in')
         - value: balance * latest_rate (numeric expression)
         - GROUP BY item, category

       Use `CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, `CREATE OR REPLACE VIEW` for idempotent re-applies.

       **Why SQL view not JS computation:** The EXACT same balance logic exists in renderer.js getBalances(). The SQL view performs the aggregation server-side, eliminating the need to transfer all raw entries to the mobile client. The view definition mirrors the JS logic identically so both sources produce the same numbers. (RESEARCH.md "Balance calculator" section: should be ported to SQL view.)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Express server → PostgreSQL | Database credentials cross network boundary (localhost TCP/5432) |
| User file system → .env | Credentials stored on disk, accessible to anyone with machine access |
| Mobile client → Express API | Untrusted network requests on local LAN (PIN auth in Plan 01-02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | pool.query() calls | mitigate | Use parameterized queries (`$1`, `$2`) via pg Pool — never string-interpolate user input into SQL (RESEARCH.md Common Pitfalls — SQL injection) |
| T-01-02 | Information Disclosure | .env file on disk | mitigate | .env added to .gitignore (Task 1); .env.example contains placeholder values only; .env not committed |
| T-01-03 | Denial of Service | Pool exhaustion (forgotten client.release) | mitigate | Prefer pool.query() for single queries (auto-release); for transactions use try/finally + client.release() (RESEARCH.md Pitfall 1) |
| T-01-SC | Tampering | npm package installs | accept | All packages (pg, bcrypt, express-rate-limit, dotenv, supertest) audited in RESEARCH.md Package Legitimacy Audit as [APPROVED] with verified registry entries and 10+ year histories |
</threat_model>
