### A) Prepare main.js for biometric integration (modify existing approach)

The architecture decision is to keep the biometric logic self-contained and import it into main.js where the `appStateChange` listener lives. Do NOT create a separate `biometric.js` file — the logic is small enough to live in `auth.js` (which already owns the auth lifecycle).

**In `mobile/src/auth.js`**, add:

1. **Add import**:
   ```javascript
   import { BiometricAuth, BiometryError, BiometryErrorType } from '@aparajita/capacitor-biometric-auth';
   ```

2. **Add constant**:
   ```javascript
   const BIOMETRIC_ENABLED_KEY = 'biometricEnabled';
   ```

3. **Add exported function `tryBiometricAuth()`** (self-contained, follows the exact pattern from RESEARCH.md Pattern 4):
   ```javascript
   export async function tryBiometricAuth() {
     // Check if biometric has been enabled (D-52)
     const { value: enabled } = await Preferences.get({ key: BIOMETRIC_ENABLED_KEY });
     if (!enabled) return { ok: false, reason: 'not_enabled' };

     // Check if biometry hardware is available (D-50/D-51)
     let biometryInfo;
     try {
       biometryInfo = await BiometricAuth.checkBiometry();
     } catch {
       return { ok: false, reason: 'unavailable' };
     }

     if (!biometryInfo.isAvailable) {
       return { ok: false, reason: 'unavailable' };
     }

     // Attempt authentication (D-51)
     try {
       await BiometricAuth.authenticate({
          reason: 'Unlock Zestok',
         cancelTitle: 'Use PIN',
         allowDeviceCredential: true,
       });

       // Success — ensure flag is set for future resume attempts (D-52)
       await Preferences.set({ key: BIOMETRIC_ENABLED_KEY, value: 'true' });
       return { ok: true };
     } catch (error) {
       if (error instanceof BiometryError) {
         if (error.code === BiometryErrorType.userCancel) {
           return { ok: false, reason: 'cancelled' };
         }
         return { ok: false, reason: 'error', code: error.code };
       }
       return { ok: false, reason: 'error' };
     }
   }
   ```

4. **Add exported function `isBiometricEnabled()`** for querying the flag:
   ```javascript
   export async function isBiometricEnabled() {
     const { value } = await Preferences.get({ key: BIOMETRIC_ENABLED_KEY });
     return value === 'true';
   }
   ```

5. **Update `clearSession()`** to also clear the biometric flag:
   ```javascript
   // Add to the existing clearSession function:
   await Preferences.remove({ key: BIOMETRIC_ENABLED_KEY });
   ```

### B) Modify main.js appStateChange handler

**In `mobile/src/main.js`:**

1. **Add to import** (from auth.js):
   ```javascript
   import { verifyPin, onAuthChange, handleSessionExpiry, checkSessionTimeout, tryBiometricAuth } from './auth.js';
   ```

2. **Update the `appStateChange` listener** at lines 199-207 — replace the existing handler with biometric-first logic (per D-51, pitfall 4):
   ```javascript
   App.addListener('appStateChange', async ({ isActive }) => {
     if (isActive) {
       // Attempt biometric first (D-51, pitfall 4)
       const bioResult = await tryBiometricAuth();

       if (bioResult.ok) {
         // Biometric succeeded — skip PIN gate entirely
         // The auth state is already set; dashboard should already be visible
         // If not visible (e.g., app was in background with gate shown), hide it
         hidePinGate();
         return;
       }

       // Biometric unavailable/cancelled — show PIN gate as fallback
       const timeout = await checkSessionTimeout();
       if (timeout.expired) {
         showSessionExpired('Session expired');
       }
       showPinGate();
     }
   });
   ```

   **Critical behavioral change from Phase 3:** The old code unconditionally called `showPinGate()` on every resume. The new code calls `hidePinGate()` only if biometric succeeds. If biometric fails or is cancelled, the PIN gate is shown. This matches D-51 exactly.

3. **Note on biometric-enabled flag (D-52):** The `biometricEnabled` flag is set to `'true'` the FIRST time biometric authentication succeeds (inside `tryBiometricAuth()`). This means:
   - Cold start: always shows PIN gate (D-52 says "once biometrics succeed for the first time")
   - First successful unlock: sets flag in `tryBiometricAuth()`
   - Subsequent resumes: flag is already `'true'`, so biometric is attempted
   - If user clears app data: flag gone, back to PIN-only

   There is NO separate enrollment screen per D-52. The `biometricEnabled` flag is set implicitly on first successful biometric use.

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-04-13 | Spoofing | biometric cancelled falls to PIN gate | mitigate | `cancelTitle: 'Use PIN'` gives user explicit option. `authenticate()` throws on cancel → captured by catch handler → returns `cancelled` → PIN gate shown. No silent bypass possible. |
