    Rewrite the existing `ipcMain.handle("stock:load", ...)` handler (currently line 67: `async () => store.read()`) to query PostgreSQL first, falling back to JSON when PG is unreachable.

    Per D-13 (PG is source of truth) and D-14 (JSON is read-only fallback).

    ```javascript
    ipcMain.handle("stock:load", async () => {
      // Check PG connectivity flag set during startup (Plan 02-01)
      if (global.pgConnected) {
        try {
          const { pool } = require('./db/pool');
          const result = await pool.query(
            'SELECT * FROM stock_entries ORDER BY created_at DESC'
          );
          return { entries: result.rows };
        } catch (err) {
          console.error("PG query failed, falling back to JSON:", err.message);
          global.pgConnected = false; // Mark PG as down for subsequent calls
        }
      }
      
      // JSON fallback (D-14: read-only fallback when PG unreachable)
      return await store.read();
    });
    ```

    **Why the `global.pgConnected` flag pattern:**
    - Set to `true` after successful `SELECT 1` in startup (Plan 02-01)
    - Set to `false` on first failed query — avoids retrying a dead connection on every load
    - The startup check (Plan 02-01) already shows a dialog if PG is down, so the user knows
    - Setting to `false` on failure enables the JSON fallback for subsequent calls without repeated PG timeouts

    **Why `pool.query()` and not a Pool client transaction:**
    - stock:load is a simple SELECT — no transaction needed (single query, auto-release)
    - RESEARCH.md Pattern 2: "Prefer pool.query() for single queries (auto-release)"

    **Column mapping:** PG `stock_entries` table uses snake_case columns (`created_at`) while the renderer expects camelCase (`createdAt`). The PG query returns snake_case keys. To maintain backward compatibility with the renderer code (which accesses `entry.createdAt`, `entry.date`, etc.), the handler must transform the column names:

    ```javascript
    const result = await pool.query('SELECT * FROM stock_entries ORDER BY created_at DESC');
    const entries = result.rows.map(row => ({
      id: row.id,
      date: row.date,             // DATE type — auto-formats to ISO string by pg
      type: row.type,
      item: row.item,
      category: row.category,
      quantity: Number(row.quantity),  // NUMERIC → number
      rate: Number(row.rate),
      note: row.note,
      createdAt: row.created_at,       // snake_case → camelCase for renderer compatibility
    }));
    return { entries };
    ```

    **Why transform to camelCase:**
    - The existing renderer code accesses `entry.createdAt` (renderer.js lines: `entry.createdAt.localeCompare(b.createdAt)` at line 130; `entry.createdAt` in the entry form at line 241)
    - The existing `createStore().read()` returns camelCase keys from JSON
    - This transformation ensures the renderer code works unchanged with PG data
    - Without transformation, all `entry.createdAt` accesses would be `undefined`, causing sort errors

    **IMPORTANT AVOIDANCES:**
    - Do NOT catch PG errors silently without logging — always log `err.message` for debugging (D-13 troubleshooting)
    - Do NOT throw from the catch block — return JSON fallback data gracefully (D-14)
    - Do NOT use `let { pool }` at module scope before dotenv — lazy-require inside handler or use module-level require after dotenv loads (avoid Pitfall 2)
    - Do NOT test for `global.pgConnected` with a bare async query if the flag is already false — respect the fallback flag
    - Do NOT use `SELECT *` in production long-term (not a concern for this phase — the stock_entries table has few columns and this is a desktop app with single-user access)
    - Do NOT change the return shape — must remain `{ entries: [...] }` matching the JSON store pattern (renderer.js init: `state.entries = Array.isArray(data.entries) ? data.entries : []`)

  <action>
    Rewrite the existing `ipcMain.handle("stock:save", ...)` handler (currently lines 69-72) to:
    1. Insert/Upsert to PostgreSQL (D-13: PG is source of truth)
    2. Always write JSON backup (D-15: auto-export on save for data safety)

    ```javascript
    ipcMain.handle("stock:save", async (_event, payload) => {
      const entries = Array.isArray(payload.entries) ? payload.entries : 
                      Array.isArray(payload) ? payload : [];
      
      // 1. Write to PostgreSQL (source of truth — D-13)
      if (global.pgConnected) {
        try {
          const { pool } = require('./db/pool');
          const client = await pool.connect();
          
          try {
            await client.query('BEGIN');
            
            for (const entry of entries) {
              await client.query(
                `INSERT INTO stock_entries (id, date, type, item, category, quantity, rate, note, created_at)
                 VALUES ($1, $2, $3, $4, $5, $6, $7, $8, $9)
                 ON CONFLICT (id) DO UPDATE SET
                   date = EXCLUDED.date,
                   type = EXCLUDED.type,
                   item = EXCLUDED.item,
                   category = EXCLUDED.category,
                   quantity = EXCLUDED.quantity,
                   rate = EXCLUDED.rate,
                   note = EXCLUDED.note`,
                [
                  entry.id,
                  entry.date,
                  entry.type,
                  entry.item,
                  entry.category || '',
                  entry.quantity,
                  entry.rate,
                  entry.note || '',
                  entry.createdAt || new Date().toISOString()
                ]
              );
            }
            
            await client.query('COMMIT');
          } catch (err) {
            await client.query('ROLLBACK');
            throw err; // Re-throw to be caught by outer catch
          } finally {
            client.release();
          }
        } catch (err) {
          console.error("PG save failed:", err.message);
          global.pgConnected = false; // Mark PG as down for subsequent calls
        }
      }
      
      // 2. Always write JSON backup (D-15 + existing behavior)
      await store.write(payload);
      
      return payload;
    });
    ```

    **Why transaction with BEGIN/COMMIT/ROLLBACK:**
    - Multiple entries are saved in one call (renderer saves the full `state.entries` array)
    - Transaction ensures atomicity: all entries succeed or none do (D-13 data integrity)
    - `ON CONFLICT (id) DO UPDATE` handles both new entries and edits

    **Why always write JSON backup (even when PG succeeds):**
    - D-15: Auto-export ensures JSON backup exists
    - D-14: JSON is the read-only fallback — it must be current for when PG is down
    - This maintains backward compatibility with the existing behavior (renderer always calls save after state change)

    **SQL injection prevention:**
    - All user-supplied values use parameterized queries (`$1`, `$2`, etc.) — never string-interpolated
    - This is the standard mitigation per RESEARCH.md Security Domain: "Use pg parameterized queries"
    - The `entry.*` fields come from the renderer which has no Node.js access (contextIsolation: true) — but defense-in-depth applies

    **IMPORTANT AVOIDANCES:**
    - Do NOT skip JSON write when PG fails — always call `await store.write(payload)` to preserve the current state (per D-15)
    - Do NOT use `pool.query()` inside the loop — use a client with BEGIN/COMMIT for atomicity (RESEARCH.md Anti-Patterns: "Forgetting client.release in transactions")
    - Do NOT forget `client.release()` in the finally block — unreleased clients deplete the pool (T-01-03 Denial of Service)
    - Do NOT assume payload is always `{ entries: [...] }` — the original handler accepted both shapes per RESEARCH.md Pattern 2 comments
    - Do NOT add validation or filtering in this handler — the renderer validates before calling save. PG constraints (CHECK type IN ('in','out'), CHECK quantity > 0, etc.) are the database-level safety net
    - Do NOT remove the existing stock:export-report-pdf handler — it remains unchanged

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Desktop renderer → Main Process (IPC) | Stock entry data crosses IPC boundary via stock:load / stock:save channels |
| Main Process → PostgreSQL | Entry data sent to PG via parameterized queries |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Tampering | SQL injection in stock:load | mitigate | Query is fully static (SELECT with no user input). Renderer cannot inject SQL — no string interpolation |
| T-02-06 | Tampering | SQL injection in stock:save | mitigate | All entry fields use parameterized queries ($1..$9). Entry values bound, never concatenated into SQL string |
| T-02-07 | Denial of Service | Client not released in transaction | mitigate | client.release() in finally block (not in catch). Research.md Anti-Patterns explicitly warns about unreleased clients |
| T-02-08 | Spoofing | JSON fallback accepts stale data | accept | JSON fallback is read-only per D-14. Staleness risk accepted — PG is the source of truth (D-13); JSON is a best-effort backup |
</threat_model>
