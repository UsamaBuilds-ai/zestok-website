    **pin:status handler** — Returns whether a PIN has been configured (D-05):
    ```javascript
    ipcMain.handle("pin:status", async () => {
      try {
        const { pool } = require('./db/pool');
        const result = await pool.query(
          "SELECT value FROM app_settings WHERE key = 'pin'"
        );
        return { configured: result.rows.length > 0 && result.rows[0].value.length > 0 };
      } catch (err) {
        console.error('PIN status check failed:', err.message);
        return { configured: false, error: 'database_unreachable' };

    });
    ```

    **pin:set handler** — Sets a new PIN (shown when no PIN exists, per D-06):
    ```javascript
    ipcMain.handle("pin:set", async (_event, { pin, confirmPin }) => {
      // Validate inputs
      if (!pin || !confirmPin) {
        throw new Error('Both PIN fields are required');
      }
      if (pin !== confirmPin) {
        throw new Error('PINs do not match');
      }
      if (!/^\d{4,6}$/.test(pin)) {
        throw new Error('PIN must be 4-6 numeric digits');
      }
      
      // Check if PIN already exists (prevent overwrite via set)
      const { pool } = require('./db/pool');
      const existing = await pool.query(
        "SELECT value FROM app_settings WHERE key = 'pin'"
      );
      if (existing.rows.length > 0 && existing.rows[0].value) {
        throw new Error('A PIN is already configured. Use change instead.');
      }
      
      // Hash and store (D-06: bcrypt hash, never plaintext)
      const hash = bcrypt.hashSync(pin, 10);
      await pool.query(
        `INSERT INTO app_settings (key, value) VALUES ('pin', $1)
         ON CONFLICT (key) DO UPDATE SET value = $1, updated_at = NOW()`,
        [hash]
      );
      return { success: true, message: 'PIN set successfully' };
    });
    ```

    **pin:change handler** — Changes existing PIN (shown when PIN exists, per D-06):
    ```javascript
    ipcMain.handle("pin:change", async (_event, { currentPin, newPin, confirmNewPin }) => {
      // Validate inputs
      if (!currentPin || !newPin || !confirmNewPin) {
        throw new Error('All PIN fields are required');
      }
      if (newPin !== confirmNewPin) {
        throw new Error('New PINs do not match');
      }
      if (!/^\d{4,6}$/.test(newPin)) {
        throw new Error('New PIN must be 4-6 numeric digits');
      }
      
      // Verify current PIN
      const { pool } = require('./db/pool');
      const result = await pool.query(
        "SELECT value FROM app_settings WHERE key = 'pin'"
      );
      if (result.rows.length === 0 || !result.rows[0].value) {
        throw new Error('No PIN configured. Use set instead.');
      }
      
      const storedHash = result.rows[0].value;
      if (!bcrypt.compareSync(currentPin, storedHash)) {
        throw new Error('Current PIN is incorrect');
      }
      
      // Hash new PIN and store
      const hash = bcrypt.hashSync(newPin, 10);
      await pool.query(
        `INSERT INTO app_settings (key, value) VALUES ('pin', $1)
         ON CONFLICT (key) DO UPDATE SET value = $1, updated_at = NOW()`,
        [hash]
      );
      return { success: true, message: 'PIN changed successfully' };
    });
    ```

    **get:local-ip handler** — Returns local network IP for mobile connection (D-07, RESEARCH.md Pattern 6):
    ```javascript
    ipcMain.handle("get:local-ip", () => {
      try {
        const interfaces = os.networkInterfaces();
        const ips = [];
        
        for (const name of Object.keys(interfaces)) {
          for (const net of interfaces[name]) {
            // Handle both string and number family types (cross-platform)
            const familyV4 = typeof net.family === 'string' ? 'IPv4' : 4;
            if (net.family === familyV4 && !net.internal) {
              ips.push({ name, address: net.address });
            }
          }
        }
        
        // Return all non-internal IPv4 addresses (let user identify the right one)
        // Per Research.md Open Question 3: show all non-internal IPv4s
        return { addresses: ips, primary: ips.length > 0 ? ips[0].address : '127.0.0.1' };
      } catch (err) {
        console.error('Local IP detection failed:', err.message);
        return { addresses: [], primary: '127.0.0.1' };
      }
    });
    ```
    **Why return all addresses (not just the first):** On Windows, virtual adapters (Docker, Hyper-V, VPN) often appear first. Showing all non-internal IPv4 addresses lets the user pick the correct one for their mobile device (RESEARCH.md Open Question 3).

    **Step 2 — Update src/preload.js with new channels:**

    Add to the existing `contextBridge.exposeInMainWorld` call:

    ```javascript
    contextBridge.exposeInMainWorld("pinApi", {
      getStatus: () => ipcRenderer.invoke("pin:status"),
      setPin: (payload) => ipcRenderer.invoke("pin:set", payload),
      changePin: (payload) => ipcRenderer.invoke("pin:change", payload),
      getLocalIp: () => ipcRenderer.invoke("get:local-ip")
    });
    ```

    The new `window.pinApi` object is separate from the existing `window.stockApi` — keeping concerns separated per codebase conventions.

    **Why bcrypt.compareSync (not async):**
    - Desktop IPC handlers run in the main process event loop. `bcrypt.compareSync` is synchronous and fast for 4-6 digit PINs (sub-millisecond). Using the async version adds unnecessary complexity for a desktop-local operation.
    - Per RESEARCH.md: "bcryptjs is pure JS, no native dependencies, same API as bcrypt."

    **Why validate PIN in main process (not just renderer) (D-08):**
    - Security boundary: the renderer is untrusted (DevTools can be opened). Main process validation is the security gate.
    - Renderer also validates for UX feedback (instant inline errors), but the main process must never trust the renderer's validation.

    **IMPORTANT AVOIDANCES:**
    - Do NOT store PIN value in any global variable in main process — read from DB on each request, write hash-only (D-06)
    - Do NOT return the hash in any IPC response — only return `{ success: true }` or error messages
    - Do NOT log the PIN value — log only generic error messages (Security Domain: "PIN verification error")
    - Do NOT use `bcrypt.compareSync` with strings longer than 72 bytes — PINs are 4-6 digits, safe, but document the guard per RESEARCH.md Pitfall 2
    - Do NOT remove the existing `stockApi` bridge — appending `pinApi` alongside it is the established pattern
    - Do NOT expose `os.networkInterfaces()` directly in preload — call via IPC to main process (contextIsolation prevents Node.js APIs in renderer per Architectural Responsibility Map)

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Desktop renderer → Main Process (IPC) | PIN values cross the IPC boundary — renderer is untrusted |
| Main Process → PostgreSQL | PIN hash stored in app_settings table over local TCP |
| Main Process → User (modal display) | Local IP displayed — non-sensitive, read-only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | PIN in renderer memory | mitigate | PIN values never stored in state object; held only in form fields during input; cleared on modal close (Pitfall 4 pattern) |
| T-02-02 | Tampering | PIN validation bypass (renderer) | mitigate | Main process re-validates all PIN operations (regex, bcrypt compare). Renderer validation is UX-only optimization |
| T-02-03 | Information Disclosure | PIN in application logs | mitigate | Never log PIN values — generic messages only ('PIN change failed: invalid current PIN') |
| T-02-04 | Elevation of Privilege | Unauthorized PIN change | mitigate | pin:change requires currentPIN verified via bcrypt.compareSync before allowing change |
| T-02-05 | Tampering | SQL injection in PIN operations | mitigate | All queries use parameterized placeholders ($1) — pin value never string-interpolated into SQL |
| T-02-SC | Tampering | npm/pip/cargo installs | mitigate | pg [SUS] requires blocking human checkpoint before install (see Plan 02-01). bcryptjs [OK], dotenv [OK] |
</threat_model>
