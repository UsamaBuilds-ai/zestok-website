<objective>
Implement offline PIN verification fallback and complete error state handling — the third vertical slice. When the user's device is offline (no network connection), PIN verification uses bcrypt.compareSync() against the stored pinHash from a previous successful online login. All error states (invalid PIN, network failure, rate-limiting) have distinct, polished UI treatments. Rate-limit errors show a 5-second timer countdown before re-enabling the input. Invalid PIN attempts trigger a shake animation. Network errors silently fall through to offline verification.

Purpose: Without offline fallback, the app is unusable in areas with poor connectivity (warehouses, stock rooms, basements). The rate-limit UX prevents abuse while giving clear feedback. Polished error states deliver professional feel required for production deployment.

Output: Updated mobile/src/auth.js (offline verify + rate-limit handling), updated mobile/src/main.js (error UI orchestration), updated mobile/src/style.css (shake animation), updated mobile/package.json (bcryptjs dependency)
</objective>

## Key Context

- D-31: Store bcrypt hash (6 rounds) as pinHash during online verify for offline fallback
- D-32: Offline verify uses bcrypt.compareSync(pin, storedHash) — D-31 ensures pinHash is available
- D-33: 429 rate-limit error blocks input for 5 seconds with timer countdown; input auto-re-enabled after 5s; no exponential backoff
- D-34: Network error silently falls through to offline verify — no error shown to user unless both fail
- D-35: Session timeout / 401 during app usage — show generic "Session expired" message (pill banner top of screen, 5s auto-dismiss)
- D-36: 401 response to any authenticated API call triggers clearSession() and re-shows PIN gate
- D-37: Do NOT distinguish between invalid PIN reasons — always show the same "Invalid PIN" error
- D-38: Error messages use fade-out transition — 0s on appear, 0.3s ease on disappear
- Rate-limit path: online verify fails with 429 → show rate-limit error → start setTimeout(5000) → show countdown → on expiry: clear error, re-enable input
- Offline path: online verify fails with network error → check offline pinHash → if exists: bcrypt.compareSync → if match: same success path (D-34: silent success) → if no match: "Invalid PIN. Try again."
- Network error + no pinHash → generic "Could not connect to server" message
- UI-SPEC provides exact error copy for each state: "Invalid PIN. Try again.", "Too many attempts. Try again in 5s", "Could not connect to server"
